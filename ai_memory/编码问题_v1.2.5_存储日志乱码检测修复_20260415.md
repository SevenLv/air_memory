# 编码问题 - v1.2.5 存储操作日志乱码 badge 不显示修复

## 标签

编码, UTF-8, PYTHONUTF8, CP1252, 乱码检测, is_garbled, SaveLog, isGarbled, save_logs, v1.2.5, Issue#37补丁

## 问题描述

v1.2.4 版本"存储操作日志"在浏览器 F12 中看到后端响应数据本身就带有乱码（`????`），
但"查询操作日志"的 API 响应中可以正常显示中文。

## 根因分析

### v1.2.4 `isGarbled()` 检测存在漏洞

CP1252 编码失败的实际输出：
```python
'中文'.encode('cp1252', errors='replace')  # → b'????' (纯 ASCII 问号)
```

v1.2.4 的检测逻辑（前端和后端均存在此问题）：
```python
# 后端：
non_ascii = [c for c in content if not c.isascii()]
if non_ascii:  # ← BUG：纯 ???? 无非 ASCII 字符，直接跳过！
    ...

# 前端：
const hasNonAscii = [...content].some((c) => c.charCodeAt(0) > 127)
if (!hasNonAscii) return false  # ← BUG：纯 ???? 在此直接返回 false！
```

结论：纯 ASCII `????` 无法被检测为乱码，导致乱码 badge 从不显示。

### save 日志 vs query 日志表现差异

| 维度 | save 日志 | query 日志 |
|------|-----------|------------|
| F12 可见性 | `"content": "????"` 顶层字段直接可见 | `results` 嵌套在 JSON 字符串中，不显眼 |
| UI 显示 | content 直接渲染文本，乱码可见 | results 只显示"N 条结果"计数，乱码不可见 |
| query 字段 | N/A | 通常为 ASCII 查询词，看起来正常 |

## 修复方案（v1.2.5）

### 1. 后端：新增 `_is_garbled()` 独立函数

```python
def _is_garbled(content: str) -> bool:
    if not content or len(content) < 2:
        return False
    question_count = content.count('?')
    question_ratio = question_count / len(content)
    has_non_ascii = any(not c.isascii() for c in content)
    # 场景一：纯 ASCII 问号（CP1252 损坏），占比 > 50%
    if not has_non_ascii and question_ratio > 0.5:
        return True
    # 场景二：混合乱码（含非 ASCII 且高问号占比）
    if has_non_ascii and question_ratio > 0.3:
        return True
    return False
```

### 2. 后端：SaveLog 模型新增 `is_garbled: bool` 字段

- 字段不存储于数据库，由 `get_save_logs()` 读取时动态计算
- API 响应 `GET /api/v1/logs/save` 中每条记录携带 `is_garbled` 字段

### 3. 前端：`isGarbled()` 采用服务端优先策略

```typescript
function isGarbled(row: SaveLog): boolean {
  // 优先信任服务端权威结果（v1.2.5+）
  if (typeof row.is_garbled === 'boolean') {
    return row.is_garbled
  }
  // 兜底：客户端检测（兼容旧版 API），同时修复纯 ???? 漏检问题
  ...
}
```
