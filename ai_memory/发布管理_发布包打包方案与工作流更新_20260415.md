# 发布管理 - 发布包打包方案与工作流更新

## 标签

发版, 发布包, zip, release.yml, 打包方案, GitHub Actions, v1.2.6, 可执行程序, frontend/dist

## 任务背景

项目经理要求将 AIR_Memory 发布内容从仅附带文档改为附带可执行程序（zip 发布包），
使用户无需 git 即可获取程序。对应文档 deploy_guide.md v1.3 已更新描述。

## 技术方案

### 打包格式选择

选择 **zip 压缩包**（而非 PyInstaller 独立可执行文件），原因：

- 项目为 Python 本机直接运行（FastAPI + uvicorn），非独立二进制，PyInstaller 不适合
- `frontend/dist/` 已预构建并提交到仓库，无需 CI 重新构建前端
- 用户只需 Python 3.11+，zip 包含所有源码 + 预构建前端即可直接运行

### zip 包结构

- 包名：`air_memory-{tag}.zip`
- 内部顶级目录：`air_memory-{tag}/`（用户解压后进入该目录即可运行）
- 包含内容：
  - `README.md`
  - `start.sh`（macOS/Linux 启动脚本）
  - `start.bat`（Windows 启动脚本）
  - `backend/src/`（后端源码）
  - `backend/pyproject.toml`（Python 包定义）
  - `backend/requirements.txt`（依赖列表）
  - `frontend/dist/`（预构建前端静态文件）
  - `doc/`（所有文档）
- 排除内容：`.git/`、`ai_rules/`、`ai_tools/`、`ai_memory/`、
  `frontend/src/`、`backend/tests/`、`backend/Dockerfile`、
  `backend/coverage.json`、`.github/`、`tests/`、`docker-compose.yml` 等开发文件

### 发布工作流更新

文件：`.github/workflows/release.yml`

变更内容：
1. 新增"构建发布包"步骤，在 GitHub Actions 运行器上执行打包脚本
2. 更新发版说明引用：`release_notes_v1.2.5.md` -> `release_notes_v1.2.6.md`
3. 在 `gh release create` 命令中附加 zip 包：`${PKG_NAME}.zip`

关键实现细节：
- 打包脚本使用 `echo "PKG_NAME=..." >> "$GITHUB_ENV"` 将变量传递给后续步骤
- zip 包在 Actions 运行时临时目录生成，不提交到 git
- `frontend/dist/` 直接从仓库检出使用（已预构建），无需 Node.js 构建步骤

## 验证结果

本地执行打包脚本验证：
- zip 包大小约 739K（合理）
- 包内结构符合 deploy_guide.md 3.1 节描述（解压后为 `air_memory-v1.2.6/`）
- 正确包含 backend/src/ 下所有源文件
- 正确包含 frontend/dist/ 预构建文件

## 工具评价

- 使用 bash 脚本直接打包（cp + zip），简单可靠，无需额外依赖
- GitHub Actions 内置 zip 命令，无需额外安装步骤
- 未使用 MCP 工具（当前任务为工作流配置，不适合 MCP）

## 注意事项

- 每次发新版本时，需同步更新 `--notes-file` 引用的发版说明文件名
- frontend/dist/ 必须在提交到仓库前已预构建，否则发布包将包含空目录
