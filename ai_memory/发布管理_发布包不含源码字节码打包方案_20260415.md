# 发布管理 - 发布包不含源码字节码打包方案

## 标签

发版, 发布包, zip, release.yml, Python, 字节码, pyc, compileall, unchecked-hash, 源码保护, v1.2.6, GitHub Actions, start.sh, start.bat

## 任务背景

项目经理要求：发布的 zip 包**不包含任何 Python 源代码**，只包含编译后的可执行程序。
前端 `frontend/dist/` 已是编译产物，无需修改。
后端 `backend/src/` 原来直接复制进发布包（包含 .py 源代码），需要改为只发布字节码。

## 确认结论

旧方案中 `.github/workflows/release.yml` 含有：
```bash
cp -r backend/src "${PKG_DIR}/backend/"
```
**zip 包确实包含 Python 源代码（.py 文件）**。

## 技术方案（Lydia 设计，Wii 验证通过）

### 方案选择：空桩 .py + unchecked-hash 字节码

**原理**：
- Python 3 的 `FileFinder` 仅当找到对应 `.py` 文件时才去 `__pycache__/` 中加载 `.pyc`
- 若直接删除 .py，Python 报 `ModuleNotFoundError`（经 Wii 第一轮验证证实）
- 解决方案：保留空桩 `.py` 文件（0字节），但使用 `--invalidation-mode unchecked-hash` 编译
  - unchecked-hash 模式：Python 加载 .pyc 时不验证 .py 内容是否匹配，直接使用字节码
  - 空 .py 文件 → Python 找到模块 → 读取 unchecked-hash .pyc → 加载真实业务逻辑

### release.yml 修改（三个步骤）

1. **新增：配置多版本 Python**（3.11/3.12/3.13，使用 `actions/setup-python@v5`）
2. **新增：编译字节码**（`compileall -j 0 -q --invalidation-mode unchecked-hash backend/src/`）
3. **修改：构建发布包**：
   - 移除 `pyproject.toml` 复制（editable install 已移除，无需此文件）
   - 改 `cp -r backend/src` + 删除 .py → `cp -r backend/src` + `truncate -s 0` 清空 .py
   - 验证断言：`PYC_COUNT > 0` 且 `PY_NONEMPTY == 0`（使用 `xargs` 截断 wc -l 空格）

### start.sh / start.bat 修改

删除各自「安装依赖」部分的 `pip install --quiet --no-deps -e backend/（\）` 行。
- editable install 需要 .py 源码才能工作
- uvicorn 通过 `--app-dir backend/src` 启动，无需 editable install

## 关键约束

- GitHub Actions 从全新 checkout 开始，不存在旧版 .pyc 缓存问题
- 覆盖 Python 3.11/3.12/3.13 三个版本，用户任意版本均可运行
- 空桩 .py 对用户透明，运行体验不变

## 注意事项

- 每次发新版本时，需同步更新 `--notes-file` 引用的发版说明文件名
- `frontend/dist/` 必须在提交到仓库前已预构建
