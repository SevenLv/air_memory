## 深度根因分析

### 问题一：为什么"查询操作日志"没有乱码？

**答：UI 层只显示结果数量，不显示内容。**

`frontend/src/views/LogsView.vue` 中的 `parseResultsSummary()` 函数：

```typescript
function parseResultsSummary(results: string): string {
  try {
    const parsed = JSON.parse(results) as unknown[]
    return `${parsed.length} 条结果`  // ← 只显示数量！绝不渲染内容
  } catch (e) {
    return '（结果解析失败）'
  }
}
```

查询日志的 `results` 字段存储的是包含记忆内容的 JSON 字符串，即使其中包含 `????`，UI 也只显示 "3 条结果"。**查询日志本身可能有 `????`，只是用户看不到。**

对比存储日志：UI 直接渲染 `row.content`，`????` 完全可见。

### 问题二：乱码是存储前还是存储后产生的？

**答：存储前已产生。`content` 到达 `log_save()` 时就已经是 `????`。**

完整数据流追踪：

```
AI 调用 save_memory("中文") → MCP HTTP/stdio 接收 → FastAPI 解析 → 
memory_service.save(content) [→ ChromaDB] 
log_service.log_save(content, memory_id) [→ SQLite]
```

`content` 在两处调用中是**同一个 Python str 对象**，ChromaDB 和 SQLite 存储的内容完全相同。  
如果 SQLite 的 `save_logs.content` 是 `????`，ChromaDB 中的文档也是 `????`。

### v1.2.4 修复的局限性

v1.2.4 在 `start.bat` 添加：
```bat
if not defined PYTHONUTF8 set "PYTHONUTF8=1"
```

**关键漏洞**：`if not defined` 检查意味着——如果用户系统环境变量中已设置 `PYTHONUTF8=0`，
或某些工具（如 Task Scheduler、CI/CD）在启动前设置了 `PYTHONUTF8=0`，此行不会生效，
导致 Python 仍以 CP1252 处理字符串，新写入的中文依然变成 `????`。

同样，`start.sh` 使用：
```bash
export PYTHONUTF8="${PYTHONUTF8:-1}"
```
Shell 默认值语法 `${var:-value}` 同样不覆盖已有值，若 `PYTHONUTF8=0` 已设置，则无效。

### v1.2.5 修复的局限性

v1.2.5 添加了 `_is_garbled()` 检测和 `is_garbled` badge，但：
1. 仍未修复 `if not defined` 的漏洞
2. `main.py` 的运行时 UTF-8 重配**仅覆盖 stdout/stderr，未覆盖 stdin**：
   ```python
   sys.stdout.reconfigure(encoding='utf-8', errors='replace')
   sys.stderr.reconfigure(encoding='utf-8', errors='replace')
   # ← sys.stdin 未重配！
   ```

### stdin 缺口的实际风险

MCP SDK (`mcp/server/stdio.py`) 在 stdio transport 模式下从 `sys.stdin` 读取 JSON-RPC 消息：

```python
# MCP SDK stdio.py 注释：
# Encoding of stdin/stdout as text streams on python is platform-dependent
# (Windows is particularly problematic), so we re-wrap the underlying binary stream
# to ensure UTF-8.
stdin = anyio.wrap_file(TextIOWrapper(sys.stdin.buffer, encoding="utf-8", errors="replace"))
```

MCP SDK **有自己的 stdin 重包装逻辑**，但这发生在 **`sys.stdin.buffer`** 层面（二进制流），  
而非 `sys.stdin` 文本流层面。  
若代码在 MCP SDK 重包装前已从 `sys.stdin`（文本模式，CP1252 编码）读取内容，  
中文会以 `errors='replace'` 变成 `?`。

## v1.2.6 修复内容

### 1. start.bat — 强制覆盖（移除 `if not defined`）

```bat
# v1.2.5（有漏洞）
if not defined PYTHONUTF8 set "PYTHONUTF8=1"
if not defined PYTHONIOENCODING set "PYTHONIOENCODING=utf-8"

# v1.2.6（修复）
set "PYTHONUTF8=1"
set "PYTHONIOENCODING=utf-8"
```

### 2. start.sh — 强制覆盖（移除 `${:-}` 默认值语法）

```bash
# v1.2.5（有漏洞）
export PYTHONUTF8="${PYTHONUTF8:-1}"
export PYTHONIOENCODING="${PYTHONIOENCODING:-utf-8}"

# v1.2.6（修复）
export PYTHONUTF8=1
export PYTHONIOENCODING=utf-8
```

### 3. main.py — 补充 stdin UTF-8 重配

```python
# v1.2.5（缺失 stdin）
if hasattr(sys.stdout, 'reconfigure'):
    sys.stdout.reconfigure(encoding='utf-8', errors='replace')
    sys.stderr.reconfigure(encoding='utf-8', errors='replace')

# v1.2.6（补充 stdin 重配）
if hasattr(sys.stdin, 'reconfigure'):
    try:
        sys.stdin.reconfigure(encoding='utf-8', errors='replace')
    except Exception:
        pass
if hasattr(sys.stdout, 'reconfigure'):
    try:
        sys.stdout.reconfigure(encoding='utf-8', errors='replace')
        sys.stderr.reconfigure(encoding='utf-8', errors='replace')
    except Exception:
        pass
```

## 技术说明

### 为何历史数据无法恢复

CP1252 编码失败产生的 `????` 是**信息熵损失**。每个汉字（3字节 UTF-8）被替换为 1 个 `?`，
原始字节信息已丢失，无法反推原始汉字。

### 新数据保护层次（v1.2.6 后）

1. **启动脚本**：`PYTHONUTF8=1` 强制覆盖 → Python 进程全局 UTF-8 模式
2. **运行时重配**：`sys.stdin/stdout/stderr.reconfigure(utf-8)` → 即使 env var 未生效也有兜底
3. **警告日志**：`sys.flags.utf8_mode == 0` 时打印 WARNING → 运维可见

### 为何 query_logs 实际上也有 `????`（只是看不见）

`query_logs.results` = `json.dumps([{"content": "????", ...}], ensure_ascii=False)`  
= `'[{"content": "????", ...}]'`（字符串中有 `????`）  
UI 调用 `parseResultsSummary()` 只返回 `"N 条结果"`，`????` 永远不会渲染。
